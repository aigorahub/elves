# Plan: Elves v1.7.0 AI-Friendly Docs and Durable Memory

## Mission

Upgrade Elves from a three-document overnight harness into a lightweight docs-in-the-loop system
that preserves reusable knowledge across runs, keeps documentation fresh during execution, and
conditions repos to be easier for future agents to navigate. The release is done when the repo's
Claude skill (`SKILL.md`), Codex skill (`AGENTS.md`), templates, durable agent docs, and human docs
all describe the same `v1.7.0` model and the branch is review-ready without requiring the user to
piece the new architecture together from conversation history.

## Scope

### In Scope

- Add a durable `learnings` layer and make it first-class in the read/update order.
- Introduce a lightweight `.ai-docs/` architecture for curated durable agent docs.
- Make documentation upkeep part of the batch loop with explicit doc triggers and doc dispositions.
- Improve the review model with evidence-first language, a bounded corrective loop, and a clear
  `PENDING-DOCS` concept distinct from merge blockers.
- Update both skill surfaces in this repo: `SKILL.md` (Claude) and `AGENTS.md` (Codex).
- Update the repo templates, README, TODO, and CHANGELOG for the `1.7.0` release.

### Out of Scope

- Skeleton generation, hydration bundles, or any heavy retrieval machinery from Command Center.
- New automation, MCP tooling, or repo-side scripts beyond small documentation scaffolding.
- Changing the "never merge" rule or replacing the PR-centric overnight workflow.
- Synchronizing installed local skill copies outside this repository as part of the PR.

## Batches

### Batch 1: Durable Memory and Agent Docs Architecture

**Tasks:**

- [ ] Finalize the `learnings` file as durable cross-run memory with clear promotion rules.
- [ ] Add `.ai-docs/manifest.md`, `.ai-docs/architecture.md`, `.ai-docs/conventions.md`, and
      `.ai-docs/gotchas.md` for the Elves repo itself.
- [ ] Update the run-memory templates so `plan`, `survival guide`, `learnings`, and
      `execution log` have crisp roles and read order.
- [ ] Add a lightweight "run digest" concept and explicit `Docs impacted/updated/promoted/deferred`
      fields where they belong.

**Acceptance criteria:**

- [ ] The repo contains the four durable `.ai-docs/*` files with non-overlapping responsibilities.
- [ ] The learnings template and run templates clearly distinguish transient run memory from
      durable reusable knowledge.
- [ ] The documentation system explicitly defines the promotion flow
      `execution log -> learnings -> curated durable docs`.

**Risk:** The biggest risk is creating overlapping doc types that look good on paper but make the
agent guess where new knowledge belongs.

## Non-Negotiables

- Update both `SKILL.md` and `AGENTS.md` when behavior changes.
- Keep the architecture intentionally lightweight: durable docs and disciplined upkeep are in scope;
  hydration/skeleton automation is not.
- Preserve the existing stage-then-launch, PR-loop, and "the user merges" philosophy unless a
  contradiction is explicitly being resolved in docs.
- Treat documentation freshness as part of done, not optional cleanup.
- The agent never merges. The PR is for the user to review and merge on return.

## Test Strategy

- **Primary gate:** Internal consistency pass across version strings, file paths, and referenced
  document names.
- **Secondary gate:** `./scripts/preflight.sh` with no git/auth failures and only acknowledged local
  environment warnings.
- **Review gate:** Read all PR comments, review threads, and checks after each push and address
  anything actionable before declaring the branch ready.
- **Known limitations:** This repo has no package-managed lint/typecheck/build/test runner; proof is
  documentation consistency plus PR review cleanliness.
